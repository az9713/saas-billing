● Here's the user story. Please review it carefully.

  ---
  User Story: Email & Password Registration

  As a new visitor, I want to create an account with my email address and a
  password, so that I can access the billing application and manage my company's
   subscription.

  ---
  Acceptance Criteria

  1. A registration form accepts three fields: email address, password, and
  password confirmation.
  2. Submitting a valid form creates a User record with the password stored as a
   bcrypt hash — never in plain text.
  3. A Tenant record is created in the same transaction as the User, and the
  User is linked to that Tenant with the OWNER role.
  4. After successful registration the user is automatically signed in and
  redirected to /dashboard.
  5. A welcome email is queued (via BullMQ + welcome template) immediately after
   account creation. Email queue failure does not prevent registration from
  succeeding.
  6. If the email already exists, the form shows a clear user-facing error. The
  response does not reveal whether the account was created via a different
  method.
  7. If the two password fields do not match, the form shows a validation error
  before any server request is made.
  8. Password must be at least 8 characters; shorter passwords are rejected with
   an explanatory message.
  9. Malformed email addresses are rejected with a validation error.
  10. No raw database error messages are returned to the client under any
  failure condition.
  11. A signed-in user visiting /register is redirected to /dashboard (existing
  middleware enforces this — must not break).
  12. Each new Tenant is created with a default FREE subscription status.

  ---
  Edge Cases

  - Email normalization: trim whitespace and lowercase before uniqueness check —
   User@Example.com and user@example.com should not produce duplicate accounts.
  - Race condition: concurrent registrations with the same email should be
  handled; the DB unique constraint is the last line of defence.
  - Welcome email failure: a Resend outage must not roll back the registration.
  BullMQ retries cover re-attempts.
  - Double-submit: form must be disabled after first submission to prevent
  duplicate accounts.
  - Rate limiting: out of scope now, but the endpoint needs it before going to
  production.

  ---
  Out of Scope

  - Email verification before login (emailVerified stays null — separate story)
  - Invite flow for existing tenants
  - OAuth / social login
  - Password reset
  - MFA, CAPTCHA
  - Soft-delete for User
  - Admin-created accounts

  ---
  Open question: Should password rules go beyond minimum length (uppercase,
  digit, special character required)? Defaulting to length-only unless you
  specify otherwise.

  ---
  Does this match what you want? Reply approved to continue to the technical
  brief, describe what to change, or reply reject to stop.

  Technical Brief: User Registration Flow

  Data Model

  No migrations needed — all models exist. The registration transaction creates
  4 records atomically:

  emailVerified: null
  - Tenant — name defaults to email local-part (e.g., alice from
  alice@example.com)
  - TenantMember — links user to tenant with role OWNER
  - Subscription — plan: FREE, status: ACTIVE

  API

  POST /api/auth/register (unauthenticated)

  ┌──────┬─────────────────────────────────────────────────────────┐
  │ Body │           { email: string, password: string }           │
  ├──────┼─────────────────────────────────────────────────────────┤
  │ 201  │ { userId: string }                                      │
  ├──────┼─────────────────────────────────────────────────────────┤
  │ 400  │ { error: string } — validation failure                  │
  ├──────┼─────────────────────────────────────────────────────────┤
  │ 409  │ { error: "An account with that email already exists." } │
  ├──────┼─────────────────────────────────────────────────────────┤
  │ 500  │ { error: "Something went wrong. Please try again." }    │
  └──────┴─────────────────────────────────────────────────────────┘

  Password-confirm is client-side only — never sent to the server.

  Service Layer

  services/auth.service.ts — registerUser({ email, password }):
  1. Normalize email (trim + lowercase)
  2. Hash password with bcryptjs (cost 12)
  3. Run prisma.$transaction([createUser, createTenant, createMember,
  createSubscription])
  4. Catch P2002 → DuplicateEmailError; catch all others → RegistrationError (no
   raw DB messages)
  5. Call sendEmail(...) in try/catch that swallows failure — email queue
  failure never rolls back registration

  Frontend (app/(auth)/register/page.tsx)

  - Controlled form: email, password, confirmPassword
  - Client-side Zod validation; password-mismatch checked client-side only
  - On submit: disable form → POST /api/auth/register → on 201:
  signIn("credentials", ...) → router.push("/dashboard")
